## List 1 — Elements already consistent across all three modules

1. **Token registry authority.** All three modules draw colors exclusively from the `index.css` CSS custom property registry. No module owns or overrides token definitions.
2. **Typography implementation pattern.** All three use raw pixel integer literals in inline styles. None reference the 16 typography scale tokens (`--io-text-4xl` through `--io-text-code-sm`) defined in `index.css`.
3. **Shared ContextMenu component.** All three use `shared/components/ContextMenu` as the sole or primary right-click context menu implementation. Visual properties for context menus are governed by the shared component.
4. **Primary button background.** All three use `var(--io-accent)` as the background of primary action buttons.
5. **Secondary button border.** All three use `1px solid var(--io-border)` as the border for secondary/outline button variants.
6. **`--io-btn-*` tokens unused.** All three modules ignore the six `--io-btn-bg`, `--io-btn-hover`, `--io-btn-active`, `--io-btn-text`, `--io-btn-secondary-bg`, `--io-btn-secondary-border` tokens defined in `index.css`.
7. **`--io-input-*` tokens largely unused.** Console and Settings ignore all five `--io-input-*` tokens; Designer's `ShapePointSelector` uses `--io-input-bg` (index.css:131) and `--io-input-border` (index.css:132). Three of the five tokens remain unused across all three modules.
8. **Focus ring suppression.** All three suppress the native browser focus ring (`outline: none`) on form inputs without applying a CSS replacement using `--io-focus-ring` or `--io-input-focus-border`.
9. **Hover via DOM mutation.** All three use `onMouseEnter`/`onMouseLeave` direct DOM style mutation (`e.currentTarget.style.*`) for at least some interactive hover states rather than CSS `:hover` rules or React state.
10. **Connection state dot pattern.** All three represent connection state (WS connection, OPC source state) with 6–8px circular dots using `var(--io-success)` for connected and `var(--io-danger)` for error.
11. **`--io-modal-bg` and `--io-modal-radius` unused.** `--io-modal-bg` and `--io-modal-radius` are ignored by all three modules; `--io-modal-backdrop` IS defined (index.css:139) and IS used in ≥6 Settings files. Console and Designer do not use it.
12. **Hardcoded primary button text.** All three hardcode the text color on accent-background buttons (`#fff`, `#09090b`, or `var(--io-text-on-accent)` which is undefined) instead of using `var(--io-accent-foreground)`. **Note 2026-05-28 (Claim B):** `shared/styles/buttons.ts` `btnPrimary` now correctly uses `var(--io-accent-foreground)`; hardcoded values persist in individual files until consumer migration pass.

## List 3 — Elements where one module is the best version others could converge on

1. **Form input centralization — Settings via `settingsStyles.ts`.** `settingsStyles.ts` defines a single named `inputStyle` object that propagates automatically to all pages that import it. Console has no equivalent; Designer's `DesignerRightPanel inputStyle` is scoped to one file and disagrees with two other input surfaces in the same module. The `settingsStyles` approach — a named export from a module-level constants file — is the pattern most easily extended across other modules without introducing a new component.

2. **StatusBadge implementation — Settings `Import.tsx` token-pair pattern.** `Import.tsx` renders status badges using `background: var(--io-success-subtle)` + `color: var(--io-success)` (and equivalent pairs for other states). This requires no color math, no string interpolation, and adapts automatically to theme changes. It outperforms Console's hardcoded rgba hex arithmetic, Designer's fully hardcoded badge colors, and Settings' own `OpcSources` hex-alpha concatenation bug.

3. **Button system naming — Settings `settingsStyles.ts`.** `btnPrimary`, `btnSecondary`, `btnDanger`, `btnSmall` are named, exported constants with clear semantic intent. Console has zero named button abstractions; Designer has `IconBtn` as a primitive but five unlabeled inline variations. Named variants provide a single place to fix token deviations (e.g., replacing `--io-text-on-accent` with `--io-accent-foreground`) across all usages simultaneously.

4. **Page-level label structure — Settings `SettingsPageLayout.tsx`.** A dedicated shared component that consistently renders page title (`<h2>`, 18px/600, `--io-text-primary`), page description (13px, `--io-text-muted`), and optional action area. Console has no equivalent (the module name is raw inline text in `index.tsx`); Designer has no equivalent across its views. `SettingsPageLayout` ensures every Settings sub-page has a consistent heading hierarchy from a single component.

5. **Menu implementation discipline — Settings.** Settings uses only `shared/components/ContextMenu` and nothing else for menus — one pattern, one source of truth for visual properties. Console adds one hand-rolled dropdown alongside it; Designer adds two custom dropdowns with divergent positioning, z-index, and shadow values. Settings demonstrates that the shared component is sufficient and that additional custom panels create maintenance surface.

6. **Form label primitives — Designer `FieldLabel` component.** `DesignerRightPanel`'s `FieldLabel` is a `<label>` element (semantically correct, unlike `<div>`-based label spreads in Console and Settings), consistently applied throughout the file, with fixed values (10px/600/uppercase/0.05em/`--io-text-muted`). It is the only reusable label primitive that uses correct HTML semantics in any of the three modules, and would require minimal adaptation to become a cross-module shared component.

7. **Destructive confirmation pattern — Settings `ConfirmDialog` usage.** Settings uses `shared/components/ConfirmDialog` for destructive actions across most pages (`Certificates`, `Email`, `BulkUpdate`, `AuthProviders`, `PointManagement`). Console implements three different inline confirmation dialogs; Designer implements per-dialog confirm flows with no shared primitive. Settings demonstrates that a shared confirmation dialog is sufficient for the full range of destructive actions in a complex module.
