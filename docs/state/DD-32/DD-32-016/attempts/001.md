## What Was Attempted

Fixed the optimistic success toast that fired on workspace creation/duplicate even when the backend returned a non-2xx response.

Root cause: `consoleApi.saveWorkspace` uses `api.put`/`api.post` which return `ApiResult<T>` — a discriminated union that resolves (never rejects) on any HTTP response, including 4xx/5xx. TanStack Query therefore always called `saveMutation.onSuccess`, even when `data.success === false`. The `onSuccess` handler did not inspect `data.success` before firing the "Workspace created" / "Workspace duplicated" toast.

Fix applied to `frontend/src/pages/console/index.tsx`:
- Added a `!data.success` guard at the top of `saveMutation.onSuccess`
- When the save fails for a create operation: rolls back the optimistic workspace addition from the Zustand store (`useWorkspaceStore.getState().deleteWorkspace(ws.id)`) and from the query cache (`queryClient.setQueryData` filter), then shows an error toast
- When the save fails for a duplicate operation: same rollback + error toast
- When the save fails for rename or generic auto-save: same error toast path as the existing `onError` handler
- When `data.success === true`: original success-toast path is unchanged — "Workspace created" and "Workspace duplicated" toasts fire only on confirmed 2xx

## Why This Attempt Succeeded

The API client (`api/client.ts`) returns `ApiResult<T>` which always resolves — it wraps 4xx/5xx into `{ success: false, error: ... }` rather than throwing. TanStack Query's mutation `onSuccess` was called for all resolved promises regardless. Adding a check on `data.success` at the start of `onSuccess` correctly routes API-level failures to the error path (rollback + error toast) and API-level successes to the original success path (success toast + query invalidation).
