## What Was Attempted

Located the workspace creation flow in `frontend/src/pages/console/index.tsx`. The `createWorkspace()` function conditionally added new workspace IDs to `pendingCreateIdsRef` only when `useApi` was true. The `saveEdit()` function (the "Done" button handler) only checked `confirmedCreateIdsRef`, which is only populated via the async API `saveMutation.onSuccess` callback. As a result, when the app is in local-storage mode (`!useApi`), no toast ever fired on Done click because neither ref was populated.

Fix: removed the `if (useApi)` guard around `pendingCreateIdsRef.current.add(ws.id)` in `createWorkspace()` so new workspace IDs are always tracked. Added a second branch in `saveEdit()` to check `pendingCreateIdsRef` when `!useApi`, firing the success toast immediately (the local save via `saveWorkspacesLocal` is synchronous, so the save is already complete when Done is clicked).

The existing API path (deferred via `confirmedCreateIdsRef`, immediate via `saveMutation.onSuccess` when editMode is already false) is unchanged.

## Why This Attempt Succeeded

The bug was that `pendingCreateIdsRef.current.add(ws.id)` was gated behind `if (useApi)`. When the app runs in local-storage mode (user not authenticated, or API query failed), `useApi` is false and the tracking never happened. `saveEdit()` only checked `confirmedCreateIdsRef` (which is only populated via the async API mutation callback), so no toast fired on Done click in the non-API path. The fix tracks all new workspace creation events unconditionally and fires the toast in `saveEdit()` for the synchronous local path.
