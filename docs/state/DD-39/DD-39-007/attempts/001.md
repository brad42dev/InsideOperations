## What Was Attempted

Implemented expression AST serialization in `export_graphic` within
`services/api-gateway/src/handlers/iographic.rs`.

Key findings from code inspection:
- Expressions are stored inline in the `bindings` JSONB column as a top-level
  `"expressions"` sub-object (not in a separate DB table).
- The existing export code extracted shapes/pipes/annotations/layers/bindings
  from the JSONB but discarded the `expressions` sub-object, hardcoding `"expressions": {}`
  in graphic.json.
- Expression AST nodes use `{ "type": "tag_ref", "point_id": "uuid" }` per doc 23.
  Binding entries that use expressions carry `"expression_key": "<key>"` (no `"point_id"`).

Changes made:
1. Added `collect_point_ids_from_expression_ast` — recursively walks expression JSON,
   finds `tag_ref` nodes with `point_id` fields, collects UUIDs for batch resolution.
2. Added `transform_expression_ast_uuids_to_tags` — recursively transforms
   `{ "type": "tag_ref", "point_id": "uuid" }` → `{ "type": "tag_ref", "tag": "FIC-101.PV" }`.
   Unresolvable UUIDs produce `<DELETED:uuid>` placeholder.
3. In `export_graphic` UUID collection block: also calls
   `collect_point_ids_from_expression_ast` on the `expressions` sub-object so those
   UUIDs are included in the batch resolution query.
4. In the JSONB extraction: also extracts `inner_expressions` alongside the other sub-fields.
5. Added `tag_based_expressions` build loop: iterates over expression entries, handles
   both wrapped (`{"ast": {...}}`) and unwrapped (flat expression value) forms.
6. Changed `"expressions": {}` in graphic.json to `"expressions": tag_based_expressions`.
