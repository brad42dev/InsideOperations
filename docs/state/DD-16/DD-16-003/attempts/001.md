## What Was Attempted

Implemented all five update-efficiency features in the data-broker service:

1. **Change-only detection**: Modified `ShadowCache::update` in `cache.rs` to return `Option<CachedValue>` (the previously stored value via DashMap's insert return value), enabling callers to compare before deciding to fan out.

2. **Deadband filtering**: In `fanout_batch` (`fanout.rs`), the returned previous value is compared against the new value. If `|new - prev| <= |prev| * deadband`, the update is suppressed. Additionally, an exact equality check (delta == 0.0 and quality unchanged) suppresses identical re-deliveries.

3. **Per-client accumulator / batching**: Replaced the immediate `try_send` in `fanout_batch` with a `PendingMap` (`DashMap<ClientId, Vec<WsPointValue>>`). `fanout_batch` is now a synchronous function that appends to this shared map. A new `run_fanout_flusher` task drains the map every `batch_window_ms` milliseconds.

4. **Config additions** in `config.rs`: Added `batch_window_ms` (default 250), `max_silence_secs` (default 60), and `fanout_deadband` (default 0.0) â€” all configurable via environment variables.

5. **Max-silence heartbeat**: New `run_heartbeat_task` in `fanout.rs`. Runs every `max_silence_secs / 2` seconds. Uses `ShadowCache::find_silent` (new method added to `cache.rs`) to find points older than the silence window and pushes their current cached value into the pending accumulator.

6. **Staggered fanout**: In `run_fanout_flusher`, clients are sorted by UUID and each gets an offset delay of `idx * (batch_window_ms / count)` ms, spreading CPU work across the batch window.

7. Updated callers (`uds.rs`, `notify.rs`) to accept `PendingMap` and `deadband` arguments and pass through. Updated `main.rs` to create the pending/last_fanout maps and spawn the flusher and heartbeat tasks.
