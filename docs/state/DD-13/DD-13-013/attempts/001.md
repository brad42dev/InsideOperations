## What Was Attempted

Fixed two files in the Log module to guard against non-array responses from `/api/logs/templates`:

1. `frontend/src/pages/log/index.tsx` (LogPage):
   - Changed the `templatesData` queryFn to return `[]` instead of throwing when `res.success` is false, and to use `Array.isArray(res.data) ? res.data : []` when success is true
   - Changed the template filter `<select>` to use `Array.isArray(templatesData) ? templatesData : []` instead of `templatesData ?? []`
   - Changed the `<TemplatesList templates={...}>` prop to use the same Array.isArray guard

2. `frontend/src/pages/log/LogSchedules.tsx` (LogSchedules):
   - Changed `schedules` derivation from `schedData?.success ? schedData.data : []` to `schedData?.success && Array.isArray(schedData.data) ? schedData.data : []`
   - Changed `templates` derivation from `tmplData?.success ? tmplData.data : []` to `tmplData?.success && Array.isArray(tmplData.data) ? tmplData.data : []`

Root cause: `?? []` and `cond ? data : []` guards only handle null/undefined, not truthy non-array objects. When the API returns a pagination wrapper object or an error shape object with `success: true` but a non-array data field, `.map()` crashes.

## Why This Attempt Succeeded

The `?? []` guard only protects against `null`/`undefined`. When the API client returns a pagination-wrapped object `{ data: [...], pagination: {...} }` as `res.data`, or any truthy non-array object, the spread/map call crashes. Replacing all guard patterns with `Array.isArray(x) ? x : []` handles all shapes: undefined, null, objects, pagination wrappers, and error bodies. The queryFn in LogPage was also changed to return `[]` directly on API failure rather than throwing, preventing TanStack Query from entering error state and instead gracefully leaving templates as an empty array.
