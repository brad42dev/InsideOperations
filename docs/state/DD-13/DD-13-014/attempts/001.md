## What Was Attempted

Fixed the Log module crash caused by unsafe `.map()` calls on API data that could be non-array values (e.g., paginated response wrappers or error objects).

**Files modified in the worktree (`frontend/src/pages/log/`):**

1. `index.tsx` — Three fixes:
   - `activeData` queryFn: Added `Array.isArray()` guards before spreading `res.data` and `res2.data` into the returned array
   - `completedData` queryFn: Added `Array.isArray()` guard on `res.data` return
   - `templatesData` queryFn: Changed `throw new Error(...)` to `return []` so API errors don't throw into ErrorBoundary; added `Array.isArray()` guard before returning
   - `TemplatesList` prop: Changed `templates={templatesData ?? []}` to `templates={Array.isArray(templatesData) ? templatesData : []}` — this was the direct crash site

2. `LogEditor.tsx` — Two fixes:
   - Added `const templateList: LogTemplate[] = Array.isArray(templateData) ? templateData : []` before calling `.find()`
   - Added `const segmentList: LogSegment[] = Array.isArray(segmentsData) ? segmentsData : []` before calling `.find()`

**Root cause**: The `?? []` null-coalescing guard only protects against `null`/`undefined`. When the API returns a 429 or error object that the client unwraps as `{ success: true, data: { error: {...} } }`, the data is a non-array object. Calling `.map()` or `.find()` on a non-null, non-undefined object that is not an array throws `TypeError: X.map is not a function`.
