## What Was Attempted

Investigated the browser crash on /log/new navigation. Used Playwright to reproduce
the crash. Discovered two distinct issues:

1. The primary crash reproducible in Playwright: The `AppShell` component in
   `frontend/src/shared/layout/AppShell.tsx` throws `ReferenceError: _setGKeyNavTarget
   is not defined` during render. This occurs when Vite's HMR (Hot Module Replacement)
   re-evaluates the AppShell module while the component is rendering — the `_setGKeyNavTarget`
   module-level const is not yet initialized in the HMR-updated module context.

2. Root cause of browser crash: `AppShell` was NOT wrapped in an ErrorBoundary in
   `App.tsx`. When AppShell throws a ReferenceError, React retries rendering 3 times
   (all fail), then the error propagates to the React root and unmounts the entire app.
   React's retry loop causes heavy CPU usage that terminates the browser tab, explaining
   Playwright's "Target page, context or browser has been closed" error.

Fix applied: Wrapped `AppShell` in `<ErrorBoundary module="App Shell">` inside
`App.tsx`. This catches the render error and shows "App Shell failed to load" instead
of crashing the browser.

Also verified: `/log/new` itself works correctly. The LogNew component renders without
errors, templates API call succeeds (6 templates), form is stable and responsive.

## Why This Attempt Succeeded

The fix is minimal and correct: added ErrorBoundary around AppShell in App.tsx.
AppShell was the only top-level component in the authenticated route tree without
an ErrorBoundary. All module-level pages (Console, Log, etc.) already had their own
ErrorBoundary wrappers, but AppShell itself did not — meaning any crash in AppShell's
render would propagate to the React root and kill the entire app.

The /log/new page itself was not the source of the crash — it is a simple component
with no Tiptap dependencies and loads correctly. The crash was triggered by AppShell
failing during HMR-driven re-evaluation of its module scope.
