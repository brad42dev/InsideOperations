## What Was Attempted

Fixed the "templates.map is not a function" / Management tab crash in the Alerts module by addressing two root causes:

1. **Type mismatch â€” backend `variables: Vec<String>` vs frontend `TemplateVariable[]`**: The backend `NotificationTemplateRow.variables` field was serializing PostgreSQL `TEXT[]` variable-name arrays as plain JSON string arrays (e.g., `["location", "area"]`). The frontend typed these as `TemplateVariable[]` (structured objects with `name`, `label`, `required`, `default_value?`). When code accessed `v.name`, `v.label`, etc., it got `undefined` since `v` was a string.

2. **Missing null guards**: `ChannelChips` called `channels.map(...)` without guarding against null/undefined. Multiple places in `SendAlertPanel` called `.filter()`, `.forEach()`, `.some()`, and `.length` on `tpl.channels` and `tpl.variables` without null guards.

3. **CreateTemplateBody / UpdateTemplateBody accepted `Vec<String>`** for `variables`, but the frontend sent structured `TemplateVariable[]` objects. This caused JSON deserialization to fail when creating/updating templates.

### Backend fix (notifications.rs):
- Changed `NotificationTemplateRow.variables` from `Vec<String>` to `Vec<JsonValue>`
- Added `var_names_to_objects()` helper: converts `["location"]` â†’ `[{"name":"location","label":"location","required":false}]`
- Added `extract_var_name()` helper: accepts either `"location"` (string) or `{"name":"location",...}` (object) and returns the name string
- Changed `CreateTemplateBody.variables` and `UpdateTemplateBody.variables` from `Option<Vec<String>>` to `Option<Vec<JsonValue>>` to accept both formats
- Updated all 5 `NotificationTemplateRow` construction sites to use `var_names_to_objects(r.get("variables"))`

### Frontend fix (index.tsx):
- `ChannelChips` prop type changed to `string[] | null | undefined`, `.map()` uses `(channels ?? []).map(...)`
- `handleTemplateSelect`: `.filter()` and `.forEach()` use `(tpl.channels ?? [])` and `(tpl.variables ?? [])`
- `hasUnfilledRequired`: uses `(selectedTemplate.variables ?? []).some(...)`
- Variable inputs render: uses `(selectedTemplate.variables ?? []).length` and `(selectedTemplate.variables ?? []).map(...)`

## Why This Attempt Succeeded

The root cause was a type mismatch between the backend (`variables: Vec<String>`, serialized as `["location"]`) and the frontend (`NotificationTemplate.variables: TemplateVariable[]`). When the Management tab rendered templates with variables, any code accessing `v.name`, `v.label` etc. got `undefined` since `v` was a plain string. The secondary issue was missing null guards on `channels` and `variables` throughout the component.

The fix normalizes the backend output to send structured `TemplateVariable`-compatible objects without requiring a DB schema change. The frontend null guards prevent future crashes even if null values somehow appear.
