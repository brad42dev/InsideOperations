## What Was Attempted

Investigated why /settings/eula showed "Access Denied" for the admin user.

Root cause confirmed: `system:configure` was absent from the original seed migration (`20260314000033_seed_tier1.up.sql`). Migration `20260323000001` added it to the `permissions` table and assigned it to the Admin role. However, UAT on 2026-03-24 still showed failure, suggesting the migration may not have been reliably applied to all environments (or the Admin JWT predated the migration).

Additionally, the Rust `Permission` enum in `crates/io-models/src/permission.rs` was missing `SystemConfigure` and `SystemCertificates` variants, creating a type parity gap between the DB/TypeScript definition and the Rust type.

Fixes applied:
1. Added `SystemConfigure` and `SystemCertificates` to the Rust Permission enum for type parity
2. Created migration `20260324000002_ensure_admin_system_configure` — an idempotent re-assertion that `system:configure` and `system:certificates` exist in the permissions table and are assigned to the Admin role (using ON CONFLICT DO NOTHING)
3. Added 3 regression tests to `frontend/src/test/rbacVisibility.test.tsx` specifically for the EULA page access guard (DD-15-014)

The frontend route guard (`PermissionGuard permission="system:configure"` at App.tsx line 1096) and the EulaAdmin page itself were already correct and complete (split-pane markdown editor in place from DD-15-007).
