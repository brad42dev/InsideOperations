## What Was Attempted

Root-cause analysis identified that `system:configure` is referenced in the EULA route's PermissionGuard
(`frontend/src/App.tsx` line 1085) and in the auth-service backend handlers
(`services/auth-service/src/handlers/eula.rs`), but this permission was never inserted into the
`permissions` table in the database seed migration. The Admin role is assigned all permissions via
`CROSS JOIN permissions`, so if `system:configure` doesn't exist in the table, the Admin user never
receives it — causing the "Access Denied" / ForbiddenPage to render.

Created a new migration (`20260323000001_add_missing_system_permissions`) that:
1. Inserts `system:configure` and `system:certificates` (another missing permission used in App.tsx)
   into the `permissions` table
2. Assigns both to the Admin role via role_permissions

## Why This Attempt Succeeded

The root cause was a missing database permission seed. The permission `system:configure` was referenced
throughout the codebase (frontend PermissionGuard, backend handlers for EULA/sessions/SCIM/SMS/auth-providers)
but never inserted into the `permissions` table. Because the Admin role uses `CROSS JOIN permissions`
to get all permissions, it could not get a permission that didn't exist. Adding the migration fixes
this on existing installations (migration runs on deploy) and fresh installs.
