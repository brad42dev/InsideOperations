## What Was Attempted

Implemented AES-256-GCM credential encryption at rest for the email-service provider configs.

Approach:
1. Added `aes-gcm`, `rand`, and `hex` workspace dependencies to `services/email-service/Cargo.toml` (base64 was already there).
2. Created `services/email-service/src/crypto.rs` with:
   - `encrypt(plaintext, key)` — generates random 12-byte nonce, encrypts with AES-256-GCM, returns base64(nonce || ciphertext)
   - `decrypt(ciphertext_b64, key)` — base64-decodes, splits nonce, decrypts, returns plaintext
   - `secret_fields(provider_type)` — maps provider type to its secret field names
   - `encrypt_provider_secrets(config, provider_type, key)` — encrypts all secret fields in-place
   - `decrypt_provider_secrets(config, provider_type, key)` — decrypts all secret fields in-place
   - `mask_provider_secrets(config, provider_type)` — replaces secret fields with `"<masked>"`
3. Updated `services/email-service/src/config.rs`:
   - Added `master_key: [u8; 32]` field to `Config`
   - Added `load_master_key()` function with three resolution tiers:
     - systemd credential file (`$CREDENTIALS_DIRECTORY/email-master-key`)
     - `IO_EMAIL_MASTER_KEY` hex env var (dev/CI)
     - Fixed named dev key with loud warning log
4. Added `mod crypto;` to `services/email-service/src/main.rs`
5. Updated `services/email-service/src/handlers/email.rs`:
   - `create_provider`: encrypts secret fields before INSERT, masks in returned row
   - `update_provider`: fetches existing provider_type from DB if needed, encrypts secret fields before UPDATE, masks in returned row
   - `list_providers`: masks secret fields in all returned rows
6. Updated `services/email-service/src/queue_worker.rs`:
   - `process_one`: decrypts provider secrets in-memory after fetching from DB, before calling `attempt_delivery`
