## What Was Attempted

Removed the deferred state-based navigation pattern (gKeyNavTarget useState + useEffect) introduced in DD-06-021 and replaced it with a direct call to `_navigateRef.current?.(path)` in the keyboard handler.

Root cause: The DD-06-021 fix introduced `gKeyNavTarget` state + a useEffect that calls `navigate(gKeyNavTarget)` followed by `setGKeyNavTarget(null)`. With `v7_startTransition: true` in BrowserRouter, calling `navigate()` inside a useEffect wraps navigation in startTransition (low-priority). Immediately calling `setGKeyNavTarget(null)` in the same effect creates an urgent (non-transition) state update that React processes before completing the startTransition navigation, causing the navigation to silently not complete.

The fix: use `_navigateRef.current?.(path)` directly from the keyboard handler. `_navigateRef.current` is assigned `navigate` in the render body (line 583), so it always holds the live navigate function. Calling navigate directly from a synchronous event handler (not via useEffect) avoids the startTransition race condition entirely.
