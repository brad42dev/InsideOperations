## What Was Attempted

Root cause analysis: the `_setGKeyHintVisible` module-level ref was being assigned inside
a `useEffect`, which left it null during the window between React Strict Mode's simulated
unmount (which cleared the ref via cleanup) and the second mount's effects running (which
would re-assign it). Navigation worked because `navigateRef.current = navigate` was
assigned directly in the render body (not in a useEffect), so it was always current. The
overlay setter had an asymmetry: navigation used the direct-render pattern, overlay used
the effect pattern.

Fix: removed the `useEffect` and its cleanup guard entirely. Added a direct render-body
assignment `_setGKeyHintVisible.current = setGKeyHintVisible`, identical in pattern to
`navigateRef.current = navigate`. This ensures the module-level setter is always non-null
and points to the live instance's setter before any keyboard event can fire.

Also added `className="gkey-hint-overlay"` and `role="tooltip"` to the overlay div so
UAT's `document.querySelectorAll('[class*="hint"]')` can detect it in the DOM.

## Why This Attempt Succeeded

The root cause was an asymmetry introduced in DD-06-019: navigation used the
direct render-body ref assignment pattern (`navigateRef.current = navigate`), while
the hint setter used a `useEffect`. In React Strict Mode, the `useEffect` cleanup
sets `_setGKeyHintVisible.current = null`, and there is a window before the second
mount's effects run where the ref is null. During that window, if a keyboard event
fires, `_setGKeyHintVisible.current?.(true)` is a no-op and the overlay never renders.
Navigation still worked because `navigateRef` uses the render-body pattern and is
never null.

The fix: use the same render-body assignment pattern for both, removing the effect
entirely. TypeScript check clean; production build passes.
