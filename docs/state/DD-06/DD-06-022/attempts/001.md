## What Was Attempted

Identified and fixed the root cause of G-key pressing causing browser crash / navigation to
`about:blank` in Playwright UAT. The bug was in the `gKeyPending` block of the `handleKeyDown`
function in `AppShell.tsx`:

When `gKeyPending.current` is `true` and an unrecognized key is pressed (one not in `G_KEY_MAP`),
the code entered the pending block, found no mapping for the key, and returned — WITHOUT calling
`e.preventDefault()`. This meant the browser's native default action for that key was allowed to
fire. In Playwright/headless browser contexts, certain keys (e.g., Enter, Space, or other navigation
keys) dispatched without `preventDefault()` could trigger browser-level navigation, causing the
page to navigate to `about:blank`.

Additionally, the original code left `gKeyPending.current = true` when an unrecognized key was
pressed, stranding the user in the pending state until the 2000ms timer fired.

Two changes were made to `frontend/src/shared/layout/AppShell.tsx`:

1. Added a module-level `_GKEY_MODIFIER_KEYS` Set (at module scope, alongside existing `_gKeyPending`
   etc.) containing modifier key names (Shift, CapsLock, etc.) that should be silently ignored while
   the G-key sequence is pending — these don't represent navigation intent and shouldn't cancel the
   sequence.

2. Rewrote the `gKeyPending` block to:
   - First: ignore bare modifier key presses (guard against cancelling sequence on Shift etc.)
   - Second: ALWAYS call `e.preventDefault()` for any non-modifier key while pending, preventing
     any native browser action from firing
   - For recognized keys (in G_KEY_MAP): navigate as before
   - For unrecognized keys: cancel the pending state and dismiss the overlay (was previously
     leaving state pending until timeout)

Note: pressing G while `gKeyPending.current = true` is correctly handled by the G-activation
block at line 833 (which comes before the pending block at line 845). The activation block
re-arms the sequence with a fresh timer. This part was already correct.

## Why This Attempt Succeeded

Root cause was the `gKeyPending` block in `handleKeyDown` failing to call `e.preventDefault()`
for unrecognized keys while in G-key pending state. This allowed the browser's native default
action for those keys to fire. In Playwright/headless Chromium, certain key events without
`preventDefault()` can cause page navigation or context destruction (manifesting as navigation
to `about:blank`).

The fix ensures `e.preventDefault()` is called for ALL non-modifier keys while `gKeyPending.current`
is true, completely preventing any native browser handling while the G-key sequence is active.
The fix also improves UX by immediately cancelling the pending state on unrecognized keys rather
than leaving the user stranded until the 2000ms timer fires.
