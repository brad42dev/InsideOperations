## What Was Attempted

Identified that the G-key hint overlay was failing to appear in the DOM after G is pressed because
React 18 automatic batching defers state updates from native `document.addEventListener` keydown
handlers. The UAT test queries the DOM synchronously after the G keypress — before React has
processed the batched state update and re-rendered the component.

Fix: added `import { flushSync } from 'react-dom'` and wrapped the `_setGKeyHintVisible.current(true)`
call (when G is pressed) in `flushSync(() => { ... })`. This forces React to flush the state update
synchronously, guaranteeing the overlay div (className="gkey-hint-overlay") is in the DOM immediately
after the G keypress returns.

The dismiss calls (setting to false) were NOT wrapped in flushSync — they don't need synchronous
updates since they don't need to be visible to DOM queries immediately.

## Why This Attempt Succeeded

Root cause: React 18 automatic batching defers state updates from native event listeners (registered
via `document.addEventListener`). When the G keypress sets `gKeyHintVisible = true`, React batches
this update and processes it asynchronously. The UAT DOM query runs synchronously after the keypress
returns, before React re-renders — so the overlay is never in the DOM from UAT's perspective.

The fix wraps the show-overlay call in `ReactDOM.flushSync(...)`, which forces React to flush
pending state updates synchronously before returning from the flushSync callback. After flushSync
completes, the overlay div is guaranteed to be in the DOM. This is the canonical React solution
for cases where DOM presence must be guaranteed immediately after a state update.

Note: the overlay was never broken in a human-visible sense — it DID render correctly for human
users because React processed the update within a few milliseconds. The failure was exclusively
in automated/programmatic DOM queries that run synchronously in the same JS tick as the keypress.
