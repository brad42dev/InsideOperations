## What Was Attempted

Fixed the `templates.find is not a function` crash in `AlertComposer.tsx`. The `queryFn` for the templates query was returning `result.data` directly without ensuring it was an array. If the API returned a paged envelope object `{ data: [...] }` or null, calling `.filter()` or `.find()` on the result would throw a TypeError.

Changes made to `frontend/src/pages/alerts/AlertComposer.tsx`:
1. Added `NotificationTemplate` to the import from `../../api/notifications`
2. Renamed the useQuery data variable from `templates` to `templatesRaw`
3. Changed the `queryFn` to normalize the result: checks `Array.isArray(d)` first, then tries to unwrap a paged envelope `{ data: [] }`, finally falls back to `[]`
4. Added `const templates: NotificationTemplate[] = templatesRaw ?? []` so `templates` is always a `NotificationTemplate[]`, never undefined or non-array
5. Removed redundant `templates &&` guards before `.filter()` and `.length` checks (now always truthy as a guaranteed array)
6. Changed `templates?.find(...)` to `templates.find(...)` (no optional chaining needed since always an array)

The `alerts/index.tsx` file was inspected and already had proper guards (`Array.isArray(result.data)` on line 340 and 1233), so no changes were needed there.
