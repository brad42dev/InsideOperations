## What Was Attempted

Root cause analysis of `TypeError: templates.find is not a function` crash on `/alerts` page load.

The `list_groups` backend handler (`services/api-gateway/src/handlers/notifications.rs`) uses `PagedResponse::new(...)` which returns `{ "success": true, "data": [...], "pagination": {...} }`. The frontend API client (`api/client.ts`) detects the `pagination` key and wraps the response as `{ data: [...], pagination: {...} }`. However, multiple frontend components treated the `listGroups()` result as a plain array by doing `result.data` directly, causing the underlying object `{ data: [...], pagination: {...} }` to be assigned to `groups` — a non-array. Calling `.find()`, `.map()`, or `.length` on this object then fails at runtime.

In a minified production bundle, the variable `groups` may appear as `templates` in the error stack trace due to name mangling.

Fixes applied:
1. `frontend/src/pages/alerts/index.tsx` — `SendAlertPanel`: changed `groups` extraction to use `Array.isArray()` detection with paginated envelope unwrapping. Added `Array.isArray()` guard to `templates` extraction.
2. `frontend/src/pages/alerts/index.tsx` — `GroupsPanel`: same paginated envelope unwrapping for groups.
3. `frontend/src/pages/alerts/AlertGroups.tsx` — changed `queryFn` to unwrap the paged envelope before returning groups data.
4. `frontend/src/pages/alerts/AlertComposer.tsx` — same fix for the groups query.
