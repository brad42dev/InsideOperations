## What Was Attempted

Three-part fix for the alert composer channels missing issue:

1. **New DB migration** (`20260324000003_alert_channels_all_enabled_seed.up.sql`): Seeds all five required channels (websocket, sms, pa, radio, push) as `enabled = true` in the `alert_channels` table. Previously only `websocket` was enabled; the other channels existed with `enabled = false` or were missing entirely.

2. **Backend rate limit bypass** (`services/api-gateway/src/mw.rs`): Added `is_static_config_endpoint()` helper that returns true for `/api/notifications/channels/enabled`. The `rate_limit` middleware now skips rate limiting for this path. Root cause: `rate_limit` runs before `jwt_auth` in the tower middleware stack, so Claims are not yet in request extensions when rate_limit executes, causing the endpoint to be classified as unauthenticated and subject to the 30 req/min IP limit (not the 600 req/min user limit). In a dev environment with hot reloads, this easily triggers 429.

3. **Frontend fallback** (`frontend/src/pages/alerts/AlertComposer.tsx`): Changed `FALLBACK_CHANNELS` from `['websocket']` to `['websocket', 'sms', 'pa', 'radio', 'push']` so the composer remains fully functional even when the API is slow or unavailable. Also fixed `handleTemplateChange` to update the `channels` state from the selected template's `channels` field (filtered to enabled channels), so the Channels section reflects the template's channel list when a template is selected.
