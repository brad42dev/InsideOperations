## What Was Attempted

Implemented Duo Security MFA (Universal Prompt via OIDC Auth API) for the auth-service.

**Approach:**
1. Added `DuoStateEntry` struct to `state.rs` — holds CSRF state token, MFA pending token reference, Duo config fields, nonce, expected username, and 10-minute expiry.
2. Added `duo_state_tokens: Arc<DashMap<String, DuoStateEntry>>` to `AppState` and initialized it in `AppState::new`.
3. Made `find_matching_key` and `jwk_to_decoding_key` in `oidc_jwks.rs` publicly accessible via wrapper functions `find_matching_key_pub` and `jwk_to_decoding_key_pub`, to support reuse from `handlers/duo.rs`.
4. Created `services/auth-service/src/handlers/duo.rs` with:
   - `duo_health_check` helper — calls `GET https://api-{hostname}.duosecurity.com/oauth/v1/health_check` with HTTP Basic auth.
   - `duo_login` handler — validates MFA pending token, loads Duo config from `auth_provider_configs` (`provider_type = 'duo'`), calls health check, generates CSRF state and nonce, stores them in `duo_state_tokens`, builds and returns Duo authorization URL.
   - `duo_callback` handler — validates CSRF state from `duo_state_tokens`, re-validates MFA pending token, exchanges code at `POST https://api-{hostname}.duosecurity.com/oauth/v1/token` with Basic auth, validates Duo ID token (custom validation: JWKS from `/.well-known/keys`, checks `iss`, `aud`, `exp`, `nonce`, `preferred_username`), consumes MFA pending token, issues I/O JWT + refresh token.
   - `verify_duo_id_token` — Duo-specific token validation (not using openidconnect crate's discovery).
5. Registered `pub mod duo;` in `handlers/mod.rs`.
6. Registered routes in `main.rs`:
   - `GET /auth/mfa/duo/callback` (static, before parameterized)
   - `GET /auth/mfa/duo/:config_id/login` (parameterized)
