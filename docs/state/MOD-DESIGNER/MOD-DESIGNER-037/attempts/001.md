## What Was Attempted

Fixed `handleNewConfirm` in `frontend/src/pages/designer/index.tsx` to save the outgoing tab's scene and viewport before calling `newDocument()`. This mirrors the save-before-switch pattern used in `openGraphicInTab()` and `switchToTab()`.

Root cause: when a user had one graphic open (Tab 1) and clicked File → New Graphic, `handleNewConfirm` called `newDocument()` which immediately replaced the active scene in sceneStore. Tab 1's `savedScene` slot remained `null` because it was never saved. When the user later clicked Tab 1 to switch back, `switchToTab()` found `savedScene=null` and fell through to the server-fetch branch — which would attempt `graphicsApi.get('new-abc123')` (a placeholder ID), causing a 404 error.

Additionally, the second tab WAS being created correctly (via `tabStoreOpenTab('new-' + crypto.randomUUID(), name)`) — but Tab 1's content was inaccessible/lost, so the UX appeared broken even though technically 2 tabs existed.
