## What Was Attempted

Three-part implementation for display element resize in the Designer canvas.

**Part 1 — getNodeBounds() fixes:**
Updated the `display_element` case in `getNodeBounds()` in `DesignerCanvas.tsx` to read actual config
dimension fields instead of hardcoded fallbacks. Each display type now reads from its correct config key:
- `text_readout`: `cfg.width ?? 92`, `cfg.height ?? 22`
- `analog_bar`: `cfg.barWidth ?? 25`, `cfg.barHeight ?? 100`
- `fill_gauge`: `cfg.barWidth ?? 24`, `cfg.barHeight ?? 80`
- `sparkline`: `cfg.sparkWidth ?? 110`, `cfg.sparkHeight ?? 18`
- `alarm_indicator`: `cfg.width ?? 24`, `cfg.height ?? 20`
- `digital_status`: `cfg.width ?? 30`, `cfg.height ?? 20`

**Part 2 — Suppress handles for SymbolInstance-child display elements:**
Added a `getNodeParent()` helper function after `boundsOverlap()`. In `SelectionOverlay`, added an
`isSymbolChildDisplayElement` check that walks the doc tree to find whether the selected display element's
parent is a `symbol_instance`. If so, `nodeShowResizeHandles` is set to false.

**Part 3 — display_element resize mouseup branch:**
Added `else if (target?.type === 'display_element')` branch after the `stencil` branch. Uses per-type
MINS table for minimum dimensions, builds a new config object via spread with the correct dimension keys,
then dispatches `CompoundCommand('Resize', [ResizeNodeCommand, ChangePropertyCommand('config', ...)])` for
single-step undo.

**Type updates in graphics.ts:**
- `TextReadoutConfig`: added `width?: number`, `height?: number`
- `SparklineConfig`: replaced literal `width: 110` / `height: 18` with `sparkWidth?: number` / `sparkHeight?: number`
- `AlarmIndicatorConfig`: added `width?: number`, `height?: number`
- `DigitalStatusConfig`: added `width?: number`, `height?: number`

Also fixed three `sparkline` default config initializers (two in DesignerCanvas.tsx, one in
DesignerRightPanel.tsx) that used the removed `width: 110, height: 18` fields.

## Why This Attempt Succeeded

All three parts implemented cleanly. The key insight was that ChangePropertyCommand uses top-level property
keys only (not dot-path), so config updates must replace the entire config object via spread
(`{ ...cfg, barWidth: finalW }`). This matches the existing pattern in DesignerRightPanel.tsx.

The SparklineConfig type had literal types `width: 110` and `height: 18` (not number) which required
renaming to `sparkWidth?: number` / `sparkHeight?: number` and updating all three initializer sites.
