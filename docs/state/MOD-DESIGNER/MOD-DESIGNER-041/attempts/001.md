## What Was Attempted

Modified the root `handleContextMenu` callback in `DesignerCanvas.tsx` to intercept right-click events in test mode. When `testMode` is true, the function now:
1. Calls `e.preventDefault()` to suppress the browser default
2. Performs a hit test to find the clicked node
3. If the hit node is a `display_element` with a `binding.pointId`, calls `setPointCtxMenu` to open the `PointContextMenu`
4. Returns early without setting `ctxNodeId`, so the Radix edit-mode context menu never opens
5. If unbound or no hit, does nothing (no menu shown)

## Why This Attempt Succeeded

Root cause was that the `handleContextMenu` callback attached to the Radix ContextMenu.Trigger (wrapping the entire canvas div) never checked `testMode`. It always called `setCtxNodeId(hitId)` which opened the edit-mode Radix context menu. The `DisplayElementRenderer`'s per-element `handleContextMenu` in the live-data branch correctly called `e.stopPropagation()` to prevent the root handler from firing, but the static (design-preview) branch had no `onContextMenu` handler at all, so the event always bubbled to the root. The fix intercepts all right-clicks at the root level in test mode, preventing the edit-mode menu from ever opening and routing bound display elements to PointContextMenu.
