## What Was Attempted

Fixed the root cause of right-click triggering drag-to-add-canvas behavior in the Designer shape palette.

Two files modified:

1. `frontend/src/pages/designer/DesignerCanvas.tsx`: Added `data-designer-canvas="true"` attribute to the canvas container div so the palette can locate it via `document.querySelector` when dispatching a center-drop for "Add to Canvas".

   - `CustomShapesPaletteTile`: Was not previously wrapped in any ContextMenu. Refactored to extract the tile JSX into `tileDiv`, added `handleAddToCanvas()`, and wrapped in `ContextMenuPrimitive.Root` with "Add to Canvas" item. Left-click drag behavior unchanged.

## Why This Attempt Succeeded

The root cause was `handleMouseDown` calling `e.preventDefault()` unconditionally before checking `e.button`. For right-clicks (button=2), this suppressed the browser's native `contextmenu` event that Radix ContextMenu relies on, and then the ghost element + drag event listeners were attached unnecessarily.

The fix (adding `if (e.button !== 0) return` at the very top of each `handleMouseDown`) allows right-clicks to pass through naturally to the Radix context menu trigger.
