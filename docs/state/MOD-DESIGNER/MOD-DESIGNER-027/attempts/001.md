## What Was Attempted

Diagnosed and fixed the root cause of the node context menu bug in DesignerCanvas.tsx.

The context menu content component `DesignerContextMenuContent` was reading `ctxNodeIdRef.current` (a plain React ref) at render time. When a node was already selected and the user right-clicked, `handleContextMenu` correctly set `ctxNodeIdRef.current = hitId` — but no React state change occurred, so `DesignerContextMenuContent` was never re-rendered. It read the stale `null` value from the previous render, causing the canvas menu branch (`nodeId === null`) to always be shown.

Fix applied:
1. Added `const [ctxNodeId, setCtxNodeId] = useState<NodeId | null>(null)` alongside the existing `ctxNodeIdRef`
2. Called `setCtxNodeId(hitId)` in `handleContextMenu` to trigger a re-render
3. Changed `DesignerContextMenuContentProps` interface: replaced `ctxNodeIdRef` with `ctxNodeId: NodeId | null`
4. Updated `DesignerContextMenuContent` function signature to accept `ctxNodeId` prop
5. Changed `const nodeId = ctxNodeIdRef.current` to `const nodeId = ctxNodeId` inside the component
6. Updated the call site to pass `ctxNodeId={ctxNodeId}` (removed `ctxNodeIdRef` prop)

## Why This Attempt Succeeded

The root cause was a React rendering timing bug: `DesignerContextMenuContent` read from `ctxNodeIdRef.current` (a plain ref) at render time. When a node was already selected, `handleContextMenu` set the ref but did not trigger a React state update, so the component was never re-rendered and read stale `null`. Adding `ctxNodeId` as React state and passing it as a prop ensures the component re-renders with the correct value whenever the context menu is triggered.
