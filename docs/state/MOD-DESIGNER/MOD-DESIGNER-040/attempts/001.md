## What Was Attempted

Fixed right-click behavior for three palette tile components that lacked context menu support:
`DisplayElementTile`, `WidgetTile`, and `ReportElementTile`. Also hardened `StencilTile`
which already had a context menu but whose `handleMouseDown` didn't guard against right-clicks.

For each tile:
1. Added `if (e.button !== 0) return` guard at the top of `handleMouseDown` so right-clicks
   do not initiate a drag/drop event that places the element on the canvas.
2. Added `handlePlaceAtCenter()` function dispatching the appropriate `io:*-drop` custom event
   targeting the canvas center.
3. Added `handleAddToFavorites()` function persisting the tile to `localStorage` under
   `io:palette-favorites`.
4. Extracted the tile div JSX to a named variable and wrapped it with
   `ContextMenuPrimitive.Root/Trigger/Portal/Content` containing "Place at Center" and
   "Add to Favorites" items.

`ShapeTile` and `CustomShapesPaletteTile` already had correct context menu implementations
and were not modified (except `StencilTile` which got the button guard).

## Why This Attempt Succeeded

The root cause was exactly as identified in the UAT notes: three tile components (`DisplayElementTile`,
`WidgetTile`, `ReportElementTile`) used `onMouseDown` without an `e.button !== 0` guard and were
not wrapped in a Radix ContextMenu. The browser `contextmenu` event was bubbling through to the
`onMouseDown` handler (which fires before `contextmenu` in some browsers) or the right-click mousedown
was triggering the drag initiation directly. The fix adds the button guard and wraps with the existing
`ContextMenuPrimitive` infrastructure already present in the file.
