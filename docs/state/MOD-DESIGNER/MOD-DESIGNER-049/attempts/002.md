## What Was Attempted

Fixed the root cause of the canvas drag-to-move creating duplicate elements bug.

Root cause: In `DesignerLeftPalette.tsx`, all 6 `handleMouseDown` functions used `document.addEventListener('mousemove/mouseup/keydown', fn, true)` (capture phase, third arg = true) but `document.removeEventListener('mousemove/mouseup/keydown', fn)` WITHOUT the capture flag (defaults to bubble = false). Per the DOM spec, `removeEventListener` must match the capture option to remove the correct listener; mismatched flags mean the listener is NEVER removed. As a result, every palette drag leaked a capture-phase `mouseup` listener on `document`, which would fire `io:display-element-drop` (or equivalent) on every subsequent `mouseup` anywhere on the page — including canvas drag-to-move interactions.

Fix: Used Python script to add `true` as the third argument to all 36 `removeEventListener` calls (6 handlers × 3 event types × 2 removal sites = 36) in `DesignerLeftPalette.tsx`.

## Files Modified

- `frontend/src/pages/designer/DesignerLeftPalette.tsx` — Added `true` (capture flag) to 36 `document.removeEventListener` calls to match their corresponding `document.addEventListener(..., true)` calls. This ensures palette drag listeners are properly removed on mouseup/escape, preventing stale capture listeners from dispatching spurious drop events during subsequent canvas interactions.
