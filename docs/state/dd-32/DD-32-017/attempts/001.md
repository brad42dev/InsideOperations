## What Was Attempted

Fixed the silent success toast on workspace creation in `frontend/src/pages/console/index.tsx`.

Root cause: After clicking "+", the backend save fires immediately (`persistWorkspace` → `saveMutation.mutate`). In fast environments the backend responds while the user is still in edit mode. `saveMutation.onSuccess` called `showToast` immediately, but the toast auto-dismissed after 5 seconds — before the user clicked Done and looked at the Notifications region. The user sees no toast.

Fix applied:
1. Added `confirmedCreateIdsRef = useRef<Set<string>>(new Set())` to track workspaces where the backend has confirmed success but the user is still in edit mode.
2. In `saveMutation.onSuccess` success path: when a create operation succeeds and `useWorkspaceStore.getState().editMode` is true, the workspace ID is placed in `confirmedCreateIdsRef` rather than showing the toast immediately.
3. `saveEdit()` now checks if `activeId` is in `confirmedCreateIdsRef` before calling `setEditMode(false)`. If it is, it deletes the ID and calls `showToast({ title: 'Workspace created', variant: 'success' })` right at Done-click time — the exact moment the user finishes creating the workspace.

Both orderings are handled:
- API responds while in edit mode → toast fires when Done is clicked
- API responds after Done is clicked (slow backend) → toast fires immediately in `onSuccess` (edit mode is false, so falls through to the existing `showToast` call)
