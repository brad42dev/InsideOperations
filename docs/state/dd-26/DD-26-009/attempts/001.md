## Why This Attempt Succeeded

The root cause was that `RecognitionWizardTrigger` had two early-return branches that rendered non-interactive `<div>` elements (not `<button>` elements) when the recognition service was unavailable (`statusError`) or both domains were disabled (`bothDisabled`). A `<div>` has no implicit button role and no click handler was attached, so clicking produced no response.

The fix removes both early-return branches, computes a `serviceUnavailableReason` string from the same status state, and always renders a `<button>` that opens the wizard. The wizard then displays the unavailability reason as a warning banner in step 1, satisfying the spec requirement to "gracefully handle the unavailable service state."
