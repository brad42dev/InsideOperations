## What Was Attempted

Fixed multi-tab support for File→New Graphic in the Designer module by making three targeted changes to `frontend/src/pages/designer/index.tsx`:

1. **handleNewConfirm**: Changed `tabStoreOpenTab('new', name)` to `tabStoreOpenTab('new-' + crypto.randomUUID(), name)`. The static `'new'` placeholder caused tabStore's idempotency check to find an existing tab with `graphicId: 'new'` and focus it rather than create a second tab.

2. **handleSave (create-new path)**: Changed the tab upgrade logic from `findTabByGraphicId('new')` (which fails for unique-ID placeholders) to using `activeTabId` — the active tab IS always the one being saved in a create-new flow. Added a `.startsWith('new-')` guard to avoid upgrading saved tabs.

3. **switchToTab restore path**: When restoring a tab from savedScene where `graphicId.startsWith('new-')`, call `useSceneStore.setState({ doc, graphicId: null, isDirty: true, ... })` instead of `loadGraphic(placeholderId, doc)`. This prevents the placeholder ID from landing in sceneStore.graphicId, which would have caused saves to attempt PUT (update) instead of POST (create).

4. **requestCloseTab restore path**: Applied the same unsaved-tab guard for the `nextTab.savedScene` branch.

## Why This Attempt Succeeded

The root cause was a static placeholder string `'new'` used as the graphicId for all new unsaved graphics. The tabStore.openTab idempotency check (`tabs.find(t => t.graphicId === graphicId)`) would find any existing `'new'` tab and focus it instead of creating a second one. Replacing this with `'new-' + crypto.randomUUID()` gives each new graphic a unique ID, allowing the tab system to create a genuinely new tab entry. The complementary fixes ensure saves and tab switches work correctly for these uniquely-IDed unsaved tabs.
