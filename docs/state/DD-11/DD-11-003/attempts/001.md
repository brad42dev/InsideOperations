## What Was Attempted

Added `export_complete` LISTEN and handler to the data-broker NOTIFY/LISTEN system so that when a report job finishes, the user who requested the job receives a targeted WebSocket `export_complete` notification.

Specific changes:

1. `crates/io-bus/src/lib.rs` — Added `ExportComplete { job_id: Uuid }` variant to `WsServerMessage` enum (alongside existing ExportNotification, ExportProgress).

2. `services/data-broker/src/notify.rs` — Refactored `run_notify_listener` to:
   - Accept two new parameters: `connections: Arc<DashMap<ClientId, mpsc::Sender<WsServerMessage>>>` and `user_connections: Arc<DashMap<Uuid, HashSet<ClientId>>>`
   - Add `listener.listen("export_complete")` after the existing `listener.listen("point_updates")`
   - Match the notification channel in the loop: `"point_updates"` goes to existing fanout logic; `"export_complete"` calls new `handle_export_complete` fn
   - `handle_export_complete` parses `{"job_id": "<uuid>"}` from payload, queries `SELECT requested_by FROM report_jobs WHERE id = $1`, then sends `WsServerMessage::ExportComplete { job_id }` to all client connections mapped to that user_id

3. `services/data-broker/src/main.rs` — Updated the `run_notify_listener` spawn to pass `Arc::clone(&connections)` and `Arc::clone(&user_connections)` (already available in scope).

4. `frontend/src/shared/hooks/useWsWorker.ts` — Updated `case 'export_complete'` handler to read `job_id` from `msg.payload?.job_id` (the nested content field from serde tag+content serialization) with fallback to `msg.job_id` for forward compatibility.
