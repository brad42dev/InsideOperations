## What Was Attempted

Created a new `expressions.rs` handler module in `services/api-gateway/src/handlers/` that implements server-side Rhai expression evaluation.

Specific changes:
1. Added `rhai.workspace = true` to `services/api-gateway/Cargo.toml`.
2. Created `services/api-gateway/src/handlers/expressions.rs` containing:
   - `ExprNode` enum with all 8 variants (Literal, PointRef, FieldRef, Unary, Binary, Function, Conditional, Group) using serde tagged enum deserialization
   - `ast_to_rhai_string()` recursive converter covering all 8 node types
   - `sanitize_identifier()` helper to produce valid Rhai variable names from arbitrary strings (e.g. OPC UA node IDs with semicolons and dots)
   - `evaluate_expression()` using `Engine::new()` with `set_max_operations(100_000)` and `eval_expression_with_scope`
   - `evaluate_expression_handler()` for `POST /api/expressions/evaluate` (ad-hoc AST)
   - `evaluate_saved_expression_handler()` for `POST /api/expressions/:id/evaluate` (saved expression via DB lookup using `sqlx::query()` runtime API)
   - 13 unit tests covering all node types and evaluation edge cases
3. Added `pub mod expressions;` to `services/api-gateway/src/handlers/mod.rs`.
4. Updated routes in `services/api-gateway/src/main.rs` — replaced `proxy_auth` stubs for the two evaluate endpoints with the new handlers.

Used `sqlx::query()` (runtime-checked) rather than `sqlx::query!()` (compile-time macro) to avoid the DATABASE_URL requirement, consistent with other handlers in the codebase.
