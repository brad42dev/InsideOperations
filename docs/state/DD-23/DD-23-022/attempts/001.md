## Why This Attempt Succeeded

Root cause: dnd-kit's default `rectIntersection` collision detection picks the droppable with the largest intersection area. When a palette tile (small button) is dragged over a container's inner DropZoneRow, the outer container sortable node and the root-zone droppable both have larger intersection areas than the inner `container-zone-{id}` div. This caused the `over.id` in `handleDragEnd` to resolve to the root-zone or container sortable ID rather than the intended `container-zone-{parentId}`.

Fix: switching to `pointerWithin` first (which picks the smallest droppable that physically contains the pointer) ensures the innermost drop target wins. The existing 3-case logic in `handleDragEnd` already handles all these cases correctly; the collision detection was the only missing piece.

Secondary fix: `handleAddFromPalette` was using `state.tiles.length` (root tile count) as the insertion index, which incorrectly positioned click-inserted tiles when the cursor was inside a container. Changed to `state.cursorIndex` which correctly tracks position within the active parent.
