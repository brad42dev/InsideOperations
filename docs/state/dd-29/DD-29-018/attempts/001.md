## What Was Attempted

Added a `useEffect` in `LockOverlay.tsx` that fires whenever `isLocked` becomes `true`. This effect calls `authApi.sessionCheck()` (GET /api/auth/me) to fetch the current `has_pin` value and updates `lockMeta` in the UI store via `setLockMeta`. The existing lock-state `useEffect` (which sets `unlockMode` based on `hasPin`) then re-runs with the fresh value.

Also added a `prevIsLockedRef` guard to the existing lock-state effect so that when it re-runs due to a `hasPin` change mid-lock (secondary run), it only updates `unlockMode` — it does NOT reset card state or set `phase = 'hidden'`, which would hide the already-visible overlay.

## Why This Attempt Succeeded

Root cause: the lock screen component read `hasPin` from `lockMeta` which was only set at app boot (via `sessionCheck()` in `AppShell.tsx`). If the user set a PIN after the app loaded, `lockMeta.hasPin` remained `false` from the initial check, so the lock screen always showed password mode.

Fix: added a `useEffect` in `LockOverlay` that re-fetches `GET /api/auth/me` whenever the session is locked (`isLocked` becomes `true`). This ensures `has_pin` is always fresh at lock time. A `prevIsLockedRef` guard prevents the secondary re-run from damaging the overlay's phase state.
