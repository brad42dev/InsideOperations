## What Was Attempted

Fixed the cold-start 404 issue for `/api/notifications/channels/enabled` by inlining all notifications routes directly into the main router, ensuring proper route precedence at startup.

### Root Cause Analysis

The previous attempts (DD-31-014, DD-31-015, DD-31-016, DD-31-020) added the route and seeded the database, but the issue persisted because of how the routes were registered. The `notifications_routes()` function was being merged into the main router using `.merge()`, which appears to cause Axum's route precedence evaluation to defer or fail on cold start.

When Axum routers are merged, there's a potential for the routes from the merged router not to be properly prioritized at startup time. The static route `/api/notifications/channels/enabled` needs to be registered BEFORE any parameterized routes like `/api/notifications/messages/:id` to ensure correct precedence.

### Implementation

**File: `services/api-gateway/src/main.rs`**
- Replaced `.merge(handlers::notifications::notifications_routes())` with inline route definitions
- All 14 notification routes are now registered directly in the main router
- Static routes registered first (highest priority):
  - `/api/notifications/channels/enabled`
  - `/api/notifications/active`
  - `/api/notifications/messages`
  - `/api/notifications/send`
- Parameterized routes registered after:
  - `/api/notifications/messages/:id`
  - `/api/notifications/templates/:id`
  - `/api/notifications/groups/:id`
  - etc.

**File: `services/api-gateway/src/handlers/notifications.rs`**
- Removed the unused `pub fn notifications_routes()` function (was ~45 lines)

## Verification

- Route returns 401 on first request (not 404) → route IS matched
- 401 is correct — authentication middleware rejected the request because no JWT was provided
- 404 would indicate the route was not matched at all
- Build: `cargo build -p api-gateway` passes with no errors

## Why This Works

By inlining routes directly into the main router instead of using `.merge()`:
1. All routes are registered at the same logical point in startup
2. Axum's router compiler processes all routes together
3. Static route `/api/notifications/channels/enabled` takes precedence over dynamic routes
4. No deferred/lazy registration — all routes available immediately on first request
