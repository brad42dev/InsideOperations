## What Was Attempted

Added a database migration to insert the `websocket` channel into the `alert_channels` table with `enabled = true`.

Root cause analysis:
- The `get_enabled_channels` handler in `services/api-gateway/src/handlers/notifications.rs` queries `SELECT channel_type FROM alert_channels WHERE enabled = true`.
- The existing seed data (`migrations/20260314000033_seed_tier1.up.sql`) only inserts `in_app` (enabled=true), `email`, `sms`, `push`, `webhook`, `teams`, `slack` (all enabled=false). There was no `websocket` entry.
- The frontend `AlertComposer.tsx` fetches enabled channels and falls back to `['websocket']` when the call fails. After DD-31-014 fixed the 404, the endpoint now returns `["in_app"]` which the frontend's `NotificationChannel` type doesn't recognize as valid.
- The fix: migration `20260323000002_alert_channels_websocket_seed` inserts `websocket` with `enabled=true` using `ON CONFLICT DO UPDATE` so it works on existing deployments too.
