## What Was Attempted

Two-part fix:

1. **Frontend (RecognitionWizard.tsx)**: Removed the `if (!statusLoaded) return null` guard that was hiding the "Recognize Image" button while the status API call was in flight. Added `statusError` state. The trigger now:
   - Always renders (never hides) when `canImport` is true
   - Shows the button in an enabled/clickable state during loading and when status is unknown
   - Shows a disabled state with explanation tooltip when the API returns an error (`statusError=true`)
   - Shows a disabled state when both domains are explicitly `mode: "disabled"` (service up but no models loaded)

2. **Backend (api-gateway/src/main.rs)**: Added a dedicated `GET /api/recognition/status` route before the wildcard `/api/recognition/*path` route. The new `get_recognition_status` handler:
   - Tries to fetch from the upstream recognition service with a 3-second timeout
   - On success: forwards the response verbatim
   - On failure (connection refused, timeout, non-2xx): returns a valid stub JSON with both domains `mode: "disabled"` and `success: true`
   - This ensures the endpoint always returns 200 with valid JSON, never 404/502

## Why This Attempt Succeeded

Root cause was two-fold:
1. The frontend trigger returned `null` while the status API call was pending (`if (!statusLoaded) return null`). In environments where the recognition service is down, the API call may return quickly with an error, but the button flickers hidden before resolving. More importantly, the spec explicitly requires the button to never be hidden due to API failure.
2. The `/api/recognition/status` endpoint returned 404/502 when the recognition service was not running, because all recognition routes were proxied directly to the service with no local fallback.

Both issues are now fixed: the button is always shown (disabled with explanation when service unavailable), and the status endpoint always returns a valid response.
