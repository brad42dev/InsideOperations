## What Was Attempted

Replaced the direct `_navigateRef.current?.(path)` call in the G-key handler with a state-driven navigation mechanism. Added a module-level `_setGKeyNavTarget` ref (same pattern as `_setGKeyHintVisible`) and a component-level `gKeyNavTarget` state. The keyboard handler now calls `_setGKeyNavTarget.current?.(path)` to set the navigation target as React state, and a new `useEffect([gKeyNavTarget, navigate])` calls `navigate(gKeyNavTarget)` when the target is set.

Root cause: calling `navigate()` directly from a native `document.addEventListener` keydown closure (even via a module-level ref) can silently no-op in React 18 concurrent mode when React Router's `v7_startTransition: true` future flag is active. This flag wraps all navigation in `startTransition`, which React can defer or cancel if called from outside the React render lifecycle. Setting state from the handler and performing navigation inside `useEffect` guarantees it runs with the live `navigate` function in the correct React context.

## Why This Attempt Succeeded

The previous fix (DD-06-019, commit 52988b2) correctly moved the navigate ref to module scope but still called it directly from the native event handler closure. With React Router's `v7_startTransition: true` future flag, `navigate()` wraps state updates in `startTransition`. When called from a native DOM event handler (outside React's event system), this can fail to commit the navigation because React schedules it as a low-priority transition that can be preempted. The new approach mirrors the proven `_setGKeyHintVisible` pattern: set React state from the handler, then perform the side effect (navigation) inside a `useEffect` that runs inside the React lifecycle with the correct navigate function bound at render time.
