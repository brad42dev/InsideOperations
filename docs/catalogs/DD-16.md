## Summary

The core broker plumbing is solid: UDS ingestion, NOTIFY/LISTEN fallback, shadow cache warm-up, subscription registry with both maps, ticket-based auth, staleness sweep, and heartbeat ping are all correctly implemented. Six gaps exist, spanning graceful shutdown (missing `server_restarting` broadcast), adaptive throttling (status_report is received and logged but triggers no throttle actions), deadband/heartbeat/staggered-fanout update-efficiency features, the per-window subscription aggregation in the SharedWorker, reconnect jitter, and immediate source-offline point-staleness marking. None of these are architectural errors — the foundations are right — but all six items are required spec behaviors.

## Findings Summary

- [DD-16-001] Graceful shutdown does not broadcast `server_restarting` or enforce 5-second drain — `src/main.rs:136`
- [DD-16-002] Adaptive throttling: `status_report` is logged but never acts on throttle state; 5-level escalation missing — `src/ws.rs:153`
- [DD-16-003] Update efficiency: no change-only fanout, no deadband, no max-silence heartbeat, no batching window, no staggered fanout — `src/fanout.rs:18`
- [DD-16-004] `point_fresh` message never emitted by broker when stale point receives new value — `src/fanout.rs` / `src/cache.rs`
- [DD-16-005] SharedWorker missing per-window subscription map, `unsubscribe-all` on port close, and reconnect jitter — `frontend/src/workers/wsWorker.ts`
- [DD-16-006] `source_offline` does not immediately mark source-associated points stale in cache — `src/uds.rs:127`
