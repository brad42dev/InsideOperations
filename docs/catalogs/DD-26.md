## Summary

The Recognition Service skeleton and Settings UI stub are in place: the service starts on port 3010, API Gateway proxies `/api/recognition/*`, the `recognition_correction` migration exists, the Settings > Recognition page uploads `.iomodel` and `.iogap` files, and the frontend API client is wired up. However, the service is deliberately a stub with no actual ONNX inference ‚ÄĒ that is expected for the current phase. Seven gaps exist: the inference endpoint path differs from the spec, several spec-mandated API endpoints are missing entirely (detect, classes, generate, feedback, model history, per-domain status), the `correction_type` column is too narrow for the longest type values, the `ort` crate and SHA-2 integrity check are absent, the Designer import wizard integration is missing, the per-domain graceful-degradation shape in `RecognitionStatus` is wrong, and the upload flow hard-codes the domain from the request body instead of auto-detecting it from `manifest.json`.
