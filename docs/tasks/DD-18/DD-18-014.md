# DD-18-014 — Add service-secret guard to archive settings endpoints

**Unit:** DD-18 (TimeSeries Data / Archive Service)  
**Wave:** 3  
**Priority:** medium  
**Source:** UAT  
**Depends-on:** []

## Problem

The archive service exposes `/settings` (GET and PUT) with no authentication guard. Any caller can read or overwrite retention and compression settings without providing the `IO_SERVICE_SECRET`. These endpoints should only be accessible to internal services.

## Spec Reference

Design doc 18 §Settings, Design doc 21 §Inter-service authentication. All intra-service admin endpoints must validate `x-io-service-secret` header against the `IO_SERVICE_SECRET` env var.

## Files to Create or Modify

- `services/archive-service/src/handlers/settings.rs` — add `check_service_secret` helper, call it in both `get_settings` and `put_settings`

## Acceptance Criteria

1. Both `GET /settings` and `PUT /settings` return `403 Forbidden` if `x-io-service-secret` header is absent or wrong.
2. Both endpoints succeed when the correct secret is provided.
3. `check_service_secret` is a private helper in the module, following the same pattern as `auth-service/src/handlers/settings.rs`.
4. `cargo check -p io-archive-service` (or `cargo check`) passes with no new warnings.

## Implementation Notes

- Follow the exact pattern from `services/auth-service/src/handlers/settings.rs` — `check_service_secret(headers, &state.config.service_secret)?`
- Add `axum::http::HeaderMap` to both handler signatures
- The `service_secret` field is already in `Config` and loaded from `IO_SERVICE_SECRET`
