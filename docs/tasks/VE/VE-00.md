## Decision File Contents

Write `docs/decisions/historical-playback-video-export.md` with the following decisions recorded:

### Format and Encoding
- Output format: WebM/VP9
- FPS range: 10–15 (default 10)
- Quality: CRF 28 (constrained quality mode, `-b:v 0`), configurable Low(35)/Medium(28)/High(22)
- Audio: none
- `-pix_fmt yuv420p` is mandatory for VLC/MPC-HC compatibility — do not drop this flag

### Capture Strategy
- Server-side headless Chromium via Playwright (Node.js capture worker)
- Dedicated minimal frontend route `/export-render` renders just the graphic at a single timestamp with no app chrome
- Playwright navigates to each timestamp URL, waits for `data-export-ready` attribute on `<body>`, screenshots, pipes PNG to FFmpeg stdin
- One frame per data step; FPS controls playback speed in the output video (not capture rate)

### Resolution
- Client reports `window.screen.width`, `window.screen.height`, `window.devicePixelRatio` at export submission time
- Server configures Playwright viewport to match; cap at 3840×2160 to prevent abuse
- SVG-based rendering scales cleanly to any resolution

### Service Placement
- New `services/video-export-service/` Rust crate on port 3011
- Rationale: isolates long-running subprocess supervision (Playwright + FFmpeg) from the api-gateway request hot path
- api-gateway proxies user-facing routes; download served directly by gateway from shared `export_dir`

### Playwright Integration
- Node.js capture worker at `services/video-export-service/capture-worker/index.mjs`
- Invoked as child process via `tokio::process::Command`
- Progress: JSON lines to stdout `{"type":"progress","frame":N,"total":M}`
- Completion: `{"type":"done","duration_seconds":...,"file_size":...}` or `{"type":"error","message":"..."}`

### Headless Browser Authentication
- Backend mints a short-lived "export ticket" JWT: `sub=user_id`, `aud="video-export"`, TTL 30min
- Contains only the user's read-tier permissions (no write/delete/admin)
- Capture worker injects it via `page.addInitScript(() => localStorage.setItem('io_access_token', ticket))`
- Rationale: user's regular access token may expire during a long render; long-lived tokens are a security hazard

### Notification
- Reuse existing `pg_notify('export_complete')` → data-broker → WebSocket → toast pipeline
- Add `kind` field to payload: `"data_export"` (existing) vs `"video_export"` (new)
- Add `status` field: `"completed"` or `"failed"` so the toast can show the right copy

### RBAC
- New permissions: `process:video_export`, `console:video_export`
- Granted to: Admin, Shift Supervisor, Operator, Analyst
- Not granted to: Content Manager, Maintenance, Viewer, Contractor (can be granted via custom roles)
- CC overlay toggle: permission-free, available to all users

### Storage and Retention
- Completed files: `${EXPORT_DIR}/videos/{job_id}.webm`
- In-flight temp files: `${EXPORT_DIR}/videos/.tmp/{job_id}/`
- Retention: reuses `EXPORT_RETENTION_HOURS` (default 24h)
- Orphan recovery: on service startup, rows stuck in `queued` or `processing` are marked `failed`

### Concurrency
- Single concurrent render per user (enforced at API layer; 409 if already in-flight)
- Single global render at a time (enforced by single-consumer Tokio channel in the service)

### Subtitle / Timestamp Overlay
- CC toggle in playback bar: state in `usePlaybackStore` (`ccOverlayEnabled: boolean`), persisted to localStorage
- When on: `<TimestampOverlay>` component rendered over the graphic in-app (as a portal)
- In exported video: if `overlay_timestamp=true` in the job params, the `/export-render` route renders the overlay component which burns into the screenshot
- Format: `YYYY-MM-DD HH:mm:ss` absolute datetime (no elapsed time)
- Style: bottom-center, semi-transparent black pill, white monospace text
